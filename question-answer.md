**模块一：Context / Performance**
2. 一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？
一个 session 聊到 200 轮，context 快爆时，不能简单把前文“摘要一下”就塞回去。真正要做的是把上下文从“逐字保存”切换成“状态保存”。也就是说，压缩的目标不是保留所有话，而是保留继续对话所必需的认知状态：用户是谁、目标是什么、已经达成了哪些共识、做过哪些决定、当前卡在哪、哪些约束不能忘、哪些细节以后可能还要追溯。

我会把压缩分成几层。第一层是会话摘要，也就是对历史对话做 rolling summary，把早期内容沉淀成一份可持续更新的“会话记忆”。这里不能写成流水账，而要写成面向后续推理的结构化摘要，比如背景、用户偏好、当前任务、已确认事实、未解决问题、关键决策、禁止事项。第二层是工作状态，也就是 task state，例如当前正在实现哪个功能、改了哪些文件、下一步应该验证什么。第三层是可检索记忆，也就是把历史中的关键片段做 embedding，放进向量库或 KV memory，后面只有在语义相关时再通过 retrieval augmented generation 拉回来，而不是每轮都带上全部历史。

压缩时最重要的是区分“事实”和“语气”。事实决定能不能接着干活，语气决定聊起来是否像同一个人。很多系统压缩后变得生硬，是因为只保留了任务摘要，丢掉了交互风格和用户偏好。比如用户不喜欢分点、希望回答像人的思维、要求专业但不要模板化，这类信息必须进入 profile memory 或 instruction memory。否则模型虽然知道任务，但说话方式会断层，用户会明显感觉“换了个人”。

具体执行上，可以采用分层上下文管理。最近几轮原文保留，因为它们承载了即时指代和情绪语境；中期内容压成 summary；长期重要信息进入 persistent memory；低价值寒暄、重复确认、已完成的中间过程可以丢弃。压缩不是一次性动作，而是随着 token 水位触发，比如达到 70% context window 时开始后台摘要，达到 85% 时进行上下文重组。这样不会等到快溢出时被迫粗暴截断。

为了确保压缩后仍然流畅，需要做摘要质量校验。一个可行做法是让压缩器输出后，再做一次 consistency check：摘要里是否保留了当前目标、用户约束、关键实体、开放问题、最近承诺；是否引入了原文没有的判断；是否把不确定信息写成了确定事实。更严格一点，可以用 entailment check 或让另一个 judge model 对照原对话做事实覆盖率评估，避免 summary drift，也就是摘要一轮轮滚动后逐渐偏离原意。

还有一个关键点是恢复机制。压缩后的上下文不应该是唯一真相，原始对话最好仍然存档。模型正常对话时使用压缩状态，但当用户追问“刚才我说的那个细节”或者任务涉及历史决策时，系统可以通过 semantic retrieval 或时间索引把相关原文片段取回。这能解决一个常见问题：摘要适合延续主线，但不适合承载所有细节。流畅感来自摘要，准确性来自可回溯的原文检索。

所以我的设计会是：短期上下文保留原文，中期上下文滚动摘要，长期信息结构化记忆，历史细节向量检索兜底。压缩后的 prompt 里不只放“之前聊了什么”，还要放“现在应该如何继续”。这样 agent 才不会像失忆后读了一页会议纪要，而是像一个真正参与过前面 200 轮对话的人，记得重点，接得住话，也不会被无关历史拖垮推理空间。

**模块二：Memory**
1. 和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？ 
我会先强调一点：用户问了以前问过的问题，并不代表 Agent 应该立刻把历史答案翻出来复读。更合理的做法是先判断这次问题是否真的需要 memory。因为用户可能只是想重新确认，也可能换了场景、换了目标，甚至之前的答案已经不适用了。所以 memory 召回的第一步不是检索，而是理解当前意图：这是一个纯知识问题，还是一个和用户过去偏好、项目状态、历史决策有关的问题。

如果判断需要召回，我会把 memory 分成几类来看。长期稳定的信息，比如用户的职业、技术栈、表达偏好，可以放在 semantic memory；某一次具体聊过什么，属于 episodic memory；用户喜欢“先结论后细节”这种工作方式，属于 procedural memory；如果是某个项目进行到哪一步，则是 task memory。用户问以前问过的问题时，通常不能只查某一轮历史对话，而要把这些信息合起来看。比如用户半个月前问过“Agent memory 怎么设计”，今天又问“长期记忆怎么做”，表面上相似，但如果今天是在准备面试，那回答结构就应该变成更适合面试表达的版本。

召回时我不会只依赖 embedding 相似度。语义相似只能说明“可能相关”，不能说明“应该使用”。我会综合考虑相关性、时间、用户是否明确标记过重要、这个话题出现过几次、是否和当前任务有关，以及旧信息是否和当前输入冲突。尤其是冲突处理很重要：当前消息优先于历史记忆，用户明确纠正优先于模型推断，新的偏好优先于旧偏好。否则 Agent 很容易显得“记性很好但不懂事”。

真正放进上下文的 memory 也应该克制。不是把半个月聊天记录全部塞给模型，而是把少量经过筛选的证据放进去，例如“用户之前讨论过 Agent memory 的分层设计”“用户现在正在准备面试”“用户偏好回答有结构但不要碎片化”。这样模型既能利用历史，又不会被大量无关上下文干扰。对于旧对话本身，可以通过摘要、关键片段和任务状态来召回，而不是直接塞完整聊天记录。

回答时也要自然。比较好的方式是轻轻带一下历史，比如“你之前也问过一个接近的问题，当时我们主要从 memory 类型来拆；这次如果放在面试场景，我会更强调召回、冲突更新和隐私控制。”这样用户会觉得 Agent 是在延续上下文，而不是在展示监控感。除非用户明确要求查历史，否则我不会暴露很细的时间戳和原文位置，比如“你在某天某点说过什么”，这会让体验变得不舒服。

最后，memory 召回还要配套更新机制和用户控制。不是每次对话都写入长期记忆，只保存稳定偏好、用户明确要求记住的信息、长期任务状态，以及对未来明显有帮助的事实。如果用户今天改变了偏好，旧记忆要被更新、降权或者废弃。同时用户应该能查看“你记得我什么”、删除某条记忆、关闭 memory，敏感信息则默认谨慎处理，必要时需要确认。

所以我的整体思路是：先判断需不需要 memory，再从多种 memory 来源中召回，经过重排和冲突判断后，只把少量高价值信息放进上下文。最终回答要以当前问题为主，历史记忆只是辅助，而不是主角。这样 Agent 才会显得既有连续性，又不过度打扰用户。

**模块三：Task**
2. 用户给 Agent 下达任务：每天早上 9 点根据昨天聊天情况做复盘总结。你会怎么设计？
我会把它设计成一个“定时触发的后台 Agent 工作流”，而不是依赖用户每天来问。用户下达任务后，系统先把这个意图转成一条可持久化的 schedule rule：每天早上 9 点执行，任务内容是基于“昨天”的聊天记录生成复盘总结。这里的关键不是会不会写总结，而是能不能稳定、准时、可追溯地运行。

到每天 9 点，Scheduler 触发一个 job，把时间窗口固定为昨天 00:00 到 23:59:59，按用户时区取数。然后从 conversation store 里拉取这段时间的会话记录，包括用户输入、Agent 回复、任务执行结果、重要工具调用和状态变化。为了避免把一整天原文直接塞进模型，可以先做 session segmentation，把对话按任务、主题或时间间隔切成几段，再分别做摘要，最后汇总成一份日复盘。

复盘本身我会分成两层生成。第一层是事实层，用来提取昨天发生了什么：用户问了哪些问题，Agent 做了哪些事情，哪些任务完成了，哪些还没完成，过程中有哪些关键决策。第二层是反思层，用来总结模式和改进点：用户真实关注点是什么，Agent 哪些地方响应得好，哪里应该更主动，哪些信息需要今天继续跟进。这样生成出来的不是聊天流水账，而是一份能帮助第二天继续工作的 operational summary。

为了让结果可靠，生成前要做 grounding。模型不能凭感觉复盘，必须基于检索到的原始聊天片段和结构化事件。可以把原始记录存在 message log，把重要动作存在 event log，再通过 RAG 或时间窗口查询取回证据。生成后再做一次 consistency check，确认总结没有编造任务、没有把未完成说成完成、没有遗漏显式承诺。对于高价值场景，还可以保留引用索引，让系统能回溯到具体消息。

交付方式也要设计清楚。9 点 job 完成后，可以把总结写入 daily review store，并通过站内通知、邮件、IM 或 App push 发给用户。如果用户没有打开应用，summary 仍然应该被保存；等用户回来时，Agent 可以自然接上：“我已经基于昨天的聊天做了复盘，今天还有两件事值得继续。”这会让 Agent 像一个持续工作的助手，而不是一次性问答工具。

最后还要考虑失败和变更。Scheduler 要有 retry、幂等 key 和执行日志，避免重复发送；用户可以修改时间、暂停任务或调整复盘风格；如果昨天没有聊天，就生成一条轻量结果，而不是硬凑内容。整体上，这个设计的核心是：用 scheduler 保证触发，用 conversation/event store 保证数据来源，用分层摘要和校验保证质量，用持久化和通知保证用户真的收到。这样每天 9 点的复盘才是一个稳定的 Agent 能力，而不是一次 prompt 技巧。

**模块四：Tool / Session Runtime**
1. Agent 工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？
我会把异步工具设计成“提交即返回，完成后续接”的模式。用户发起任务时，Agent 不直接阻塞等待工具跑完，而是先创建一个 job，把任务参数、用户上下文、会话 id、幂等 key、回调目标写入任务队列，然后立刻告诉用户：“这个任务需要后台处理，我已经开始执行，完成后会通知你。”这样用户不会被卡在一个长时间 loading 里，但系统也没有把任务丢掉。

执行层上，异步工具不应该由对话进程自己跑，而应该交给独立的 worker。对话进程负责意图识别和任务编排，worker 负责真正执行耗时操作，比如爬取、生成报告、批量处理文件、调用慢 API。任务状态放在 job store 里，至少要有 pending、running、succeeded、failed、cancelled 这些状态，同时记录进度、重试次数、错误原因和产物地址。这样用户中途回来问“刚才那个任务怎么样了”，Agent 可以查状态，而不是凭记忆回答。

完成通知要分两层。第一层是系统通知，比如站内信、IM、邮件、push 或 webhook，用来告诉用户任务已经结束。第二层是会话续接，也就是把异步结果重新注入 Agent 的上下文，让 Agent 能基于结果继续说人话，而不是只甩一个下载链接。比如报告生成完成后，Agent 应该能总结关键结论，并问用户是否要继续分析某个部分。这里需要一个 completion callback 或事件总线，让 worker 完成后发布 ToolJobCompleted 事件，再由 Agent runtime 消费这个事件。

关键是结果要可追溯。异步工具完成后，不能只把结果文本发出去，还要把原始输出、结构化结果、执行日志和关联会话都存起来。通知里展示的是人能读懂的摘要，后台保留的是可审计的数据。这样一来，如果用户几小时后再回来追问，Agent 可以通过 job_id 找回工具结果，并把它作为 grounding context 使用，避免重新执行，也避免模型凭空补细节。

失败场景也要当成一等公民处理。异步任务可能超时、第三方接口失败、权限失效或结果为空。系统应该支持 retry policy、timeout、dead letter queue 和错误分类。可恢复错误可以自动重试，不可恢复错误要给用户一个明确通知，说明失败原因和下一步选择。这里最怕的是任务悄悄失败，用户以为 Agent 还在处理，信任会很快掉下去。

为了不打扰用户，通知策略也要克制。短任务可以在会话里直接回填，长任务才走后台通知；低重要性任务可以合并成摘要，高重要性任务即时推送；如果用户当前在线，就优先在当前会话续接，如果离线，再走 push 或邮件。最终设计的核心是：对话线程保持轻，异步任务有状态，完成结果能回到上下文里，用户既不用一直等，也不会错过真正重要的结果。

**模块五：Agent Runtime 架构对比**
1. Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？
Claude Code 更像是一个“面向工程过程的 Agent Runtime”，而 GLM、豆包这类 OpenAI-compatible function calling 更像是“模型 API 里的工具调用协议”。两者表面上都是模型调用工具，但抽象层级不一样。Claude Code 关心的是一次真实开发任务怎么推进：读文件、改文件、跑命令、观察输出、继续决策；OpenAI-compatible function calling 关心的是模型如何以结构化 JSON 形式声明“我要调用哪个函数、参数是什么”。

Claude Code 的工具输出通常是对 Agent 可读、对人也可观察的执行结果。比如 Bash、Read、Edit、Grep 这类工具会把命令结果、文件片段、错误信息、diff 或执行状态返回给 Agent，Agent 再基于这些观察继续行动。它的设计重点是闭环：模型不是只生成一个函数调用，而是在“计划、执行、观察、修正”的循环里工作。因此工具输出往往带有运行时语义，例如是否成功、输出被截断、是否需要权限、文件是否被修改。这种方式适合复杂软件工程，因为 Agent 需要不断根据真实环境反馈调整策略。

GLM、豆包等 OpenAI-compatible function calling 更偏协议化。模型回复里一般会出现 tool_calls 或类似字段，里面包含函数名和 JSON arguments；业务系统拿到后自己执行函数，再把结果作为 tool message 回填给模型。这里工具本身不属于模型平台的“工程运行时”，而是开发者自己定义的外部能力。优点是标准、轻量、容易接入现有 OpenAI SDK 和网关；缺点是它只定义了“调用格式”，不替你解决任务调度、权限控制、文件上下文、长任务状态、工具输出压缩、失败恢复这些 Agent Runtime 问题。

所以 Claude Code 的优势在于工程任务的端到端体验强。它把工具调用、上下文管理、权限确认、终端输出、文件编辑这些东西组织成一个连续工作流，Agent 能像开发者一样边看边改。缺点也明显：它更重、更绑定特定运行环境，工具语义和宿主系统耦合较深，不像普通 function calling 那样容易迁移到任意业务后端。

OpenAI-compatible function calling 的优势是通用性强。只要模型能产出稳定的 JSON schema，后端就可以把它接到查订单、发消息、查数据库、调用内部 API 等各种业务系统里。它也更适合做平台化，因为协议边界清楚：模型负责选择工具和生成参数，业务服务负责执行和返回结果。但它的短板是“只给了接口，没有给 Agent 操作系统”。如果要做 Claude Code 那种复杂 Agent，还得自己补 job store、event log、权限模型、tool result summarization、retry、observability 和 memory。

本质差异可以这样理解：Claude Code 的工具输出是 Agent 工作过程的一部分，强调“环境反馈”；GLM、豆包这类 function calling 是模型 API 的一段结构化输出，强调“函数调用契约”。前者更适合代码 Agent、运维 Agent、长链路任务；后者更适合业务 Agent、客服 Agent、插件式能力接入。真正做架构时不应该把它们看成谁替代谁，而是看你要的是一个完整的执行环境，还是一个可嵌入自己系统的工具调用协议。
